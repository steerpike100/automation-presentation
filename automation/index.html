Road to Automated Testing - Paxport

Introduction & aims

My background - a generalist tester with a more recent focus on automation. Previous companies: Orange, Play.com, National Trust, Dyson

We have a mixed ability group at Paxport in terms of technical abilities. The primary aim of this session is to help demystify automated testing and to generate discussion and enthusiasm. By the end of today, I hope you will have a broader understanding of how automated testing can help support what we already do. I also hope we can find some ways of working and supporting each other more closely

Some topics for discussion:

  • why would we use automation?
  • tools we can use to help us
  • how we are already using those tools to test FAB
  • how we are already using those test Fabrix
  • do I have to be involved in this process?
  • how can I learn more if I want to?

Test Team within Paxport

We have a mixed ability group at Paxport in terms of technical abilities.

  • some members are exclusively manual in their focus
  • often, the manual testers are more valuable to us (product knowledge, attention to detail, industry knowledge)
  • some have more technical expertise but limited coding experience
  • some have an enthusiasm to learn more about automation

Nobody will be asked to move quickly out of their comfort zone, but their is a desire to increase Test Team experience of automation

I am not an expert, I am certainly not the oracle; but it iss healthy to be challenged and I need others to add their ideas. There will be always be better, more efficient ways of doing things. Test Automation is often about writing test code. In this respect, we should have the same aspirations as the Developers in respect of technical governance e.g.

  • code reviews
  • best practice guidelines
  • standardisation of practices
  • buddy systems

What do we know about automated testing?

Pros

  • it can be more reliable
  • can be drastically faster
  • can take huge effort out of regression testing
  • can be used to drive the development when used in conjunction with a process like BDD and Cucumber

Cons

  • removes the 'hueristic', human element
  • places faith in good automated tests (false positives?)
  • technical overheads in training team - long lead time to change culture
  • technical overhead in maintaining dependencies
  • can be used to drive the development when used in conjunction with a process like BDD and Cucumber

Where are we now - Fabrix?

Fabrix

Fabrix is our Front End product which we sell branded, colourwashed and styled for many companies

browser automation is not easy for a number of reasons:

  • browsers behave differently
  • different websites use different technologies
  • we have to contend with latency and synchronisation issues

We have a large number of websites to try and automate - more than 50 at a guess. This is not easy when they all look different - if elements for one website are not present in the DOM of another.

Where do we aspire to be?

We need to improve technical implementations:

  • refactor code to provide a data factory from which we can build better bespoke data objects for testing
  • remove the volume of TestNG XML files
  • use Cucumber to derive Scenarios and pass in data in form of data tables
  • increase scope of the tests to support such events as FaB deployments

We need to improve technical skills, enthuse, support the team:

  • provide training
  • provide sandbox areas for playing with automated tools
  • buddy system
  • regular forums / meetups / confluence documentation
  • online training materials Udemy

We need to understand BDD

  • use BDD, in anger, as a project methodology
  • evangelise Cucumber and show how such tools can involve the *whole* team in building the software right & building the right software
  • continue building a continuous integration pipeline; get better quaility software out, faster
  • show how Cucumber help Dev and Test contribute, collaboratively to 'Acceptance Tests' - to ensure the business get the software they want and we leave a legacy of regression tests

And finally......

  • we need to find more bugs, faster and cruciually....before they impact the customer (and our revenue)
